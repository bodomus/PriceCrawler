# Task: Document CrawlerOptions and QueueOptions

## Goal

Добавить русскоязычные описания для всех полей/свойств классов:

- `CrawlerOptions`
- `QueueOptions`

Описания должны быть добавлены прямо в исходные `.cs` файлы классов, а также вынесены в отдельные Markdown-файлы:

- `[solution]/docs/CrawlerOptions.md`
- `[solution]/docs/QueueOptions.md`

## Context

Проект: PriceCrawler / Varus.

Сейчас в проекте есть option-классы конфигурации, которые используются для настройки crawler-логики и очереди обработки URL/товаров. Нужно сделать эти настройки понятными для разработчика и для будущей документации.

### 2. Добавить XML documentation comments в `.cs` файлы

Для каждого public-свойства в `CrawlerOptions` и `QueueOptions` добавить XML-комментарий на русском языке:

```csharp
/// <summary>
/// Русское описание назначения параметра.
/// </summary>
public int SomeOption { get; set; }
```

Описание должно объяснять:

- что означает параметр;
- где он используется;

- есть ли связь с другими параметрами.

Если свойство уже имеет комментарий — не удалять без причины, а улучшить/расширить его на русском языке.

### 5. Формат `docs/CrawlerOptions.md`

Файл должен быть на русском языке.

Рекомендуемая структура:

```markdown
# CrawlerOptions

## Назначение

`CrawlerOptions` содержит настройки, управляющие поведением crawler-а: выбором источника URL, ограничениями на количество обрабатываемых товаров, режимами обхода, задержками, лимитами и другими параметрами выполнения.

## Где используется

Класс используется через options/configuration binding из `appsettings*.json` и передаётся в сервисы crawler-а через `IOptions<CrawlerOptions>`, `IOptionsSnapshot<CrawlerOptions>` или аналогичный механизм, если он уже используется в проекте.

## Поля

| Поле | Тип | Описание |
|---|---:|---|
| `PropertyName` | `type` | Описание на русском языке. |

## Пример конфигурации

```json
{
  "Crawler": {
    "PropertyName": "value"
  }
}
```

## Примечания

- Не увеличивать лимиты без понимания нагрузки на сайт-источник.
- Для тестовых запусков использовать небольшие значения лимитов.
- Для production/stage окружения значения должны быть согласованы с политикой throttling/retry.
```

Codex должен заменить `PropertyName`, `type` и пример на реальные свойства класса.

### 6. Формат `docs/QueueOptions.md`

Файл должен быть на русском языке.

Рекомендуемая структура:

```markdown
# QueueOptions

## Назначение

`QueueOptions` содержит настройки очереди обработки URL/товаров: размер батча, ограничения повторов, задержки, правила выборки элементов из очереди и другие параметры, влияющие на планирование crawler-задач.

## Где используется

Класс используется через options/configuration binding из `appsettings*.json` и применяется в сервисах, которые читают, заполняют или обрабатывают очередь.

## Поля

| Поле | Тип | Описание |
|---|---:|---|
| `PropertyName` | `type` | Описание на русском языке. |

## Пример конфигурации

```json
{
  "Queue": {
    "PropertyName": "value"
  }
}
```

## Примечания

- Размер батча должен соответствовать допустимой нагрузке.
- Retry-настройки должны защищать crawler от бесконечной обработки проблемных URL.
- Queue-настройки не должны дублировать бизнес-логику crawler-а.
```

Codex должен заменить `PropertyName`, `type` и пример на реальные свойства класса.

## Documentation style requirements

Описания должны быть:

- на русском языке;
- короткими, но полезными;
- без воды;
- технически точными;
- понятными человеку, который впервые открыл проект.

Плохой пример:

```csharp
/// <summary>
/// Максимум.
/// </summary>
public int MaxItems { get; set; }
```

Хороший пример:

```csharp
/// <summary>
/// Максимальное количество URL, которое crawler может взять в обработку за один запуск.
/// Используется как защитный лимит, чтобы тестовый или stage-запуск случайно не обработал весь каталог.
/// </summary>
public int MaxUrls { get; set; }
```
