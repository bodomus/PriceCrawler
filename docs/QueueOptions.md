# QueueOptions

## Назначение

`QueueOptions` содержит настройки очереди обработки URL: размер батча, polling, lease, количество попыток, retry backoff и reaper для зависших элементов.

## Где используется

Класс привязывается из секции `Queue` в `appsettings*.json` через `IOptions<QueueOptions>`. Его читает `RunCrawlerUseCase` при постановке URL в очередь, резервировании батчей, расчете повторов и восстановлении элементов с истекшим lease.

## Поля

| Поле | Тип | Описание |
|---|---:|---|
| `BatchSize` | `int` | Максимальное количество элементов очереди, резервируемых за один цикл. Больший батч снижает число обращений к базе, но дольше удерживает lease на группе URL. |
| `PollDelayMs` | `int` | Задержка между опросами очереди в миллисекундах, когда нет доступных элементов. Меньшее значение быстрее подхватывает работу, но чаще обращается к базе. |
| `LeaseSeconds` | `int` | Срок lease для зарезервированного URL в секундах. Должен покрывать типичное время обработки, иначе возможны повторные резервации. |
| `MaxAttempts` | `int` | Максимальное количество попыток обработки одного URL до перевода в dead. Увеличение помогает transient-ошибкам, но дольше держит проблемные URL в очереди. |
| `RetryBaseDelayMs` | `int` | Базовая задержка перед повторной обработкой URL в миллисекундах. Используется при расчете backoff после ошибки. |
| `RetryMaxDelayMs` | `int` | Максимальная задержка перед retry в миллисекундах. Ограничивает рост backoff после нескольких неудачных попыток. |
| `ReaperIntervalSeconds` | `int` | Интервал запуска reaper-а в секундах. Reaper возвращает в работу элементы с истекшим lease, если worker не завершил обработку. |

## Пример конфигурации

```json
{
  "Queue": {
    "BatchSize": 32,
    "PollDelayMs": 250,
    "LeaseSeconds": 90,
    "MaxAttempts": 3,
    "RetryBaseDelayMs": 1000,
    "RetryMaxDelayMs": 30000,
    "ReaperIntervalSeconds": 15
  }
}
```

## Примечания

- `BatchSize`, `LeaseSeconds` и `MaxConcurrency` из `CrawlerOptions` нужно согласовывать: большой batch при коротком lease повышает риск повторной обработки.
- Retry-настройки должны ограничивать бесконечную обработку проблемных URL и не создавать всплески нагрузки.
- Reaper нужен для восстановления зависших элементов, но слишком частый reaper увеличивает количество запросов к базе.
